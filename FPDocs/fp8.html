Progress on Goals:

  • Tucker: Built a search dialog for selecting flora
    • The search dialog (which can be seen here) allows the user to search for any flora in the dataset, either by common name or scientific name. Originally, this was going to be a tabbed layout, allowing the user to swtich between a "search by common name" tab and a "search by scientific name" tab. We felt that combining them would be a much more elegant solution.
    • If the user clicks on one of the flora, it will become highlighted green to show that it is selected. It will also appear in the right pane, which holds the flora that have been selected so far. If the user either clicks on the highlighted flora on the left, or clicks on it on the right, the flora will be removed from the right pane and also become unhighlighted on the left.
    • The steps for creating this were as follows:
      1. Get a basic HTML/CSS structure to put everything in (i.e. properly styled div containers)
      2. Research to see find some examples of people doing something like this
      3. Using the parsed dataset that we've created already, get a list of all the flora to appear
      4. Make it so that when a user clicks a flora, it highlights, but also print out the name of it so that we know that's working
      5. Get the search to work, so that as the user is typing a word, the results start to filter
      6. Add in "no results" when there are no flora that match the search
      7. Build a second, completely hidden list that will become the selected flora list
      8. Sync up the two lists, so that whenever a flora is highlighted on the left, it will appear on the right. Then also implement functionality on both sides for removing them.
  • Sophie: Built choropleth of Maine by town based on selected samples
    • The choropleth was implemented as the most efficient way to signal to us that our map visualization was dynamically working with our dataset. It can be seen here. The response is based on an array that will be created by the search dialog, which hold all samples (with their unique scientific name so we can easily look them up in the dataset) currently selected by the user. The town element fill varies based on the relative number of samples taken there. The success of this visualization will (ideally later this week) facillitate drawing dots on each town whose size will vary with number of samples.
    • The only current interaction is for clicking is for our team. If a town is clicked on, the code prints the town name and the number of samples in the town to the console. This function only exists for debugging.
    • The steps for creating this were as follows:
      1. Research examples of people implementing maps of states by towns using d3 and topojson.
      2. Learn how and implement converting shape files from government websites into geojson files.
      3. Go through previous group project to learn how my group members implemented the reading in of information to our choropleth and how they then drew it, as I had a minimal role in this component.
      4. Create a placeholder array of some of the scientific names in our parsed dataset, which the code will later have the equivalent of when the search dialog is merged. This array represents flora the user selected.
      5. Create an dictionary, where the keys are all towns that are going to be drawn on the map, and the values, which are all initially set to 0, will be the number of samples of the current flora selection that were taken in each town.
      6. Create a function that counts how many samples of the current flora selection were taken in each town and updates the dictionary accordingly.
      7. Re-learn how the quantile function works.
      8. Draw choropleth, using the town dictionary to determine fill.
      9. Create click function to make sure counters are being modified correctly.
